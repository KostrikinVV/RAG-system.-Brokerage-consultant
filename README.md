# 🏦 RAG-система «Банковский консультант» по брокерским документам

## 📌 Описание проекта

Проект представляет собой вопросно-ответную систему (RAG — Retrieval-Augmented Generation), способную консультировать клиентов по документам брокерского обслуживания ПАО Сбербанк. Система извлекает релевантные фрагменты из базы знаний с помощью гибридного поиска (векторное сходство + BM25) и генерирует ответ, опираясь исключительно на предоставленный контекст. Реализована поддержка диалоговой памяти и кеширование ответов для работы под высокой нагрузкой.

RAG-пайплайн спроектирован как самодостаточный сервис и одновременно как промежуточный этап для перехода к агентной архитектуре (Agentic RAG) с возможностью маршрутизации запросов и подключения дополнительных инструментов.

## 🛠 Технологический стек

- **Обработка документов:** LangChain, NLTK
    
- **Векторная база данных:** Qdrant (локальное хранение)
    
- **Эмбеддинги:** `intfloat/multilingual-e5-large` (1024‑мерные векторы)
    
- **Ключевой поиск:** Rank-BM25 + лемматизация (pymorphy3)
    
- **Генеративная модель (LLM):** Ollama + `qwen3:4b`
    
- **Метрики и оценка:** pandas, numpy, Jaccard similarity (триграммы)
    
- **Оптимизация:** in‑memory кеш, батчевая загрузка в Qdrant

## 📊 Этапы реализации и архитектурные выводы

### Этап 1. Подготовка данных и чанкинг

Документы с официального сайта Сбербанка (раздел «Документы и условия по брокерскому обслуживанию») были преобразованы в текстовый формат. При выборе стратегии чанкинга были протестированы три подхода:

|Стратегия|Количество чанков|Качество|
|---|---|---|
|`CharacterTextSplitter` (жёсткая нарезка по символам)|2008|❌ Разрывает заголовки и списки|
|`NLTKTextSplitter` (по предложениям)|1954|❌ Отрывает нумерацию от пунктов|
|**`RecursiveCharacterTextSplitter`**|**2344**|✅ Сохраняет абзацы и структуру документов|

**Вывод:** Рекурсивный чанкинг (`chunk_size=600`, `chunk_overlap=150`) обеспечивает наилучшую сохранность логических блоков, что критически важно для юридических и финансовых текстов.

### Этап 2. Эмбеддинги и векторная база

Для векторизации использована модель `intfloat/multilingual-e5-large` (1024 измерения). Выбор обусловлен:

- отличной поддержкой русского языка,
    
- высокой размерностью вектора, улучшающей качество семантического поиска.
    

Все чанки загружены в локальную коллекцию Qdrant с метрикой косинусного расстояния. Для ускорения работы данные сохраняются на диск и переиспользуются между сессиями.

### Этап 3. Гибридный поиск (Retrieval)

Реализован комбинированный поиск, объединяющий:

- **Векторный поиск** (Qdrant) — семантическая близость,
    
- **BM25** (Rank-BM25 с русской лемматизацией и фильтрацией стоп‑слов) — ключевое соответствие.
    

Ранжирование производится по формуле Reciprocal Rank Fusion (RRF):

text

final_score(d) = 1/(k + rank_vector) + 1/(k + rank_bm25)

где `k=60` — сглаживающий параметр. Итоговый список формируется по убыванию `final_score`.

**Оценка качества:**  
Для объективной метрики был создан тестовый датасет из **79 вопросов**. Вопросы и эталонные ответы сгенерированы с помощью DeepSeek по следующему промпту:

> *«Я хочу выполнить оценку поисковика чанков в RAG системе. Разделение на чанки я делал по абзацам `RecursiveCharacterTextSplitter(chunk_size=600, chunk_overlap=100)`. Проанализируй документ и сформируй 20 запросов (по 10 вопросов на каждый документ), которые могут быть заданы пользователями. При формировании запросов учти, что используется гибридный поиск (семантика + BM25). К каждому запросу приложи абзац из документа без изменений, содержащий ответ.»*

Результаты на тестовой выборке:

|Метрика|Значение|
|---|---|
|**Hit Rate@10**|0.59 (59%)|
|**MRR**|0.423|

_Сравнение проводилось через нормализацию текста и триграммный коэффициент Жаккара (порог 0.5)._

### Этап 4. Интеграция с LLM и память диалога

В качестве генеративной модели используется локальный сервер Ollama с моделью `qwen3:4b`. Реализована диалоговая память: история последних двух реплик учитывается при формировании промпта и расширении поискового запроса для уточняющих вопросов.

Системный промпт жёстко ограничивает модель:

- отвечать только на основе предоставленного контекста,
    
- при отсутствии информации честно сообщать об этом и рекомендовать горячую линию.
    

Пример диалога показывает, что бот корректно поддерживает контекст и не выдумывает факты.

### Этап 5. Оптимизация производительности (кеширование)

Внедрён простой in‑memory кеш для ответов на повторяющиеся запросы. Это позволяет избежать повторной генерации LLM и поиска по базе.

**Результаты замера:**

|Режим|Время ответа|
|---|---|
|Без кеша (полный цикл: поиск + генерация)|~117 сек|
|**С кешем**|**~0.001 сек**|

Ускорение составило **более 116 000 раз**, что демонстрирует эффективность кеширования для снижения нагрузки на вычислительные ресурсы.

## 🚀 Вектор развития (Agentic Architecture)

Текущий RAG-пайплайн уже обёрнут в функцию, готовую к использованию в качестве инструмента (`Tool`) для агентных фреймворков. Следующие шаги улучшения:

1. **Маршрутизация запросов (Router Agent).** Автоматическое определение, куда направить вопрос: в локальную базу знаний, к внешнему API (например, калькулятор доходности) или на эскалацию оператору.
    
2. **Динамическое переформулирование запроса (Query Rewriting).** Для устранения проблемы «утечки контекста» при резкой смене темы диалога планируется внедрить LLM-переформулировщик, который будет очищать историю или генерировать автономный поисковый запрос.
    
3. **Расширение базы знаний.** Добавление табличных документов и приложений в формате PDF с помощью OCR и специализированных парсеров.
    
4. **Улучшение метрик поиска.** Тонкая настройка весов гибридного ранжирования и параметра `k` в RRF на основе дополнительного тестирования.
    
5. **Потоковая генерация (streaming).** Реализация постепенной выдачи ответа для улучшения пользовательского опыта.
